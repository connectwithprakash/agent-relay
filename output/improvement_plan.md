# Agent Relay - Prioritized Improvement Plan

Generated by 4-agent review team (security_auditor, performance_eng, ux_reviewer, lead).

---

## P0 - Critical / Must-Fix

| # | Finding | Source | Fix |
|---|---------|--------|-----|
| 1 | CORS wildcard (`*`) combined with `allow_credentials=True` | security_auditor | Remove wildcard origin or disable credentials. Use an explicit allowlist of trusted origins. |
| 2 | WebSocket endpoint has no authentication | security_auditor | Require API key verification on WebSocket connect handshake (query param or first message). |
| 3 | SSRF via webhook URL registration | security_auditor | Whitelist `https` scheme only, block private/internal IP ranges (169.254.x.x, 10.x.x.x, 127.x.x.x, etc.). |
| 4 | Legacy relays with no API key bypass all auth | security_auditor | Remove the `api_key_hash is None` bypass (main.py:151). Require API key for all write operations. |
| 5 | Synchronous SQLAlchemy in async endpoints | performance_eng | Migrate to `AsyncSession` + `async_engine` from `sqlalchemy.ext.asyncio`. Combine with item 10 (detached ORM fix) since they share the same refactor. |

## P1 - Important

| # | Finding | Source | Fix |
|---|---------|--------|-----|
| 6 | N+1 query in `list_relays` | performance_eng | Replace per-relay `count_by_relay_id()` loop with a single `GROUP BY` query or SQL subquery. |
| 7 | No message size limits | security_auditor | Add `max_length` constraint to `SendMessageRequest.content` and `data` fields (e.g., 64KB). |
| 8 | Webhook amplification vector | security_auditor | Rate-limit webhook registrations per relay. Cap max webhooks per relay (e.g., 5). Add timeout and response-size limits on delivery. |
| 9 | httpx client created per webhook delivery | performance_eng | Create a shared `httpx.AsyncClient` with connection pooling at app startup. Inject into webhook service. |
| 10 | Detached SQLAlchemy objects in fire-and-forget tasks | performance_eng | Create a new DB session inside the background task rather than reusing the request session. |
| 11 | API key not displayed on relay creation | ux_reviewer | Show the API key prominently in the `CreateRelayPage` success view alongside the existing `ShareLink` component. |
| 12 | MCP tool errors are raw HTTP exceptions | ux_reviewer | Catch `httpx` errors in MCP server and translate to actionable messages (e.g., "Not your turn" instead of `403 Forbidden`). Use SDK's existing typed exception classes. |

## P2 - Nice-to-Have

| # | Finding | Source | Fix |
|---|---------|--------|-----|
| 13 | SDK dual `api_key` confusion | ux_reviewer | Deprecate the per-method `api_key` override parameter. Use only the constructor-level key. |
| 14 | No `wait_for_turn` helper in SDK | ux_reviewer | Add a `wait_for_turn(relay_id, agent, poll_interval)` convenience method to both sync and async clients. |
| 15 | ConnectionManager unbounded memory | performance_eng | Delete relay key from `active_connections` dict when WebSocket list becomes empty. |
| 16 | No relay deletion or expiry in UI | ux_reviewer | Add TTL/expiry support to relays and expose delete/archive actions in the frontend. |
| 17 | SHA-256 for API key hashing | security_auditor | Consider migrating to bcrypt or argon2id. Low urgency since API keys are high-entropy. |
| 18 | Missing composite DB indexes | performance_eng | Add indexes on `(relay_id, created_at)` for messages and `(relay_id, agent_index)` for webhooks. |
| 19 | MCP `is_public` description misleading | ux_reviewer | Update docstring to match actual behavior ("accessible with direct link" vs "only owner can read"). |
| 20 | No pagination bounds on query params | security_auditor, performance_eng | Cap `limit` query parameter to a reasonable max (e.g., 100). |

---

## Implementation Notes

- **Items 5 and 10** should be tackled together as a single async-session refactor (per performance_eng).
- **Item 11** is a quick win: the `CreateRelayPage` success view already has the `ShareLink` component; just add the `api_key` from the creation response (per ux_reviewer).
- **Item 12** is also a quick win: the SDK already has typed exception classes that the MCP server can use directly (per ux_reviewer).
- **P0 items 1-4** are security vulnerabilities that should be fixed before any public deployment.
