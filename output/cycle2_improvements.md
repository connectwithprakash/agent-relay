## Cycle 2 New Findings

### CRITICAL (blocks production)

| # | Finding | Reviewer | Detail |
|---|---------|----------|--------|
| C1 | Race condition in send_message | backend_reviewer | Message creation and turn advance are two separate commits. Concurrent requests can both pass validate_turn before either commits. Needs SELECT FOR UPDATE or atomic read-check-write. |
| C2 | Turn deadlock on agent crash | agentic_reviewer | If an agent crashes mid-turn, the relay deadlocks forever. Need configurable turn timeouts with auto-advance or a skip API endpoint. |
| C3 | WebSocket reconnect loop | frontend_reviewer | useWebSocket includes onMessage/onOpen/onClose in connect useCallback deps. Inline arrow functions from RelayDashboard trigger infinite disconnect/reconnect cycles. Fix: useRef for callbacks or useCallback at call site. |
| C4 | Webhook uses stale db session in background task | backend_reviewer | trigger_webhooks is called via asyncio.create_task with the request-scoped db session. The task runs after response is sent and session may be closed. Confirmed overlap with Cycle 1 item 10. |
| C5 | No React error boundary | frontend_reviewer | Zero error boundaries in the app. Any runtime error in RelayDashboard/MessageList crashes the entire app with a white screen. |

### IMPORTANT (next sprint)

| # | Finding | Reviewer | Detail |
|---|---------|----------|--------|
| I1 | Duplicate messages from WebSocket | frontend_reviewer | useMessages.addMessage appends unconditionally. If sending agent receives its own message back via WS before send resolves, or WS reconnects and replays, duplicates appear. Need dedup by message ID. |
| I2 | No structured message protocol | agentic_reviewer | Messages are plain text or opaque dicts. Agents need typed schemas (task_request, task_result, error, handoff). The type field only allows "text" or "structured" - too coarse. |
| I3 | No wait_for_turn SDK method | agentic_reviewer | Agents must poll in a loop. SDK needs wait_for_turn(relay_id, agent, timeout) using WebSocket or long-poll. Async client has listen() but no sync equivalent or turn-specific blocking helper. |
| I4 | Service layer half-adopted | backend_reviewer | RelayService.create_relay does db.add/commit directly bypassing RelayRepository. send_message uses MessageRepository. RelayService.get_relay_state queries db directly instead of using MessageRepository. |
| I5 | list_webhooks has no auth | backend_reviewer | GET /relays/{relay_id}/webhooks has no authentication. Anyone can see all webhook URLs for any relay, leaking potentially sensitive callback URLs. |
| I6 | Missing test coverage | backend + frontend | Backend: no tests for webhook delivery, retry logic, WebSocket endpoint, concurrent sends. Frontend: only 3 of 14 components/hooks tested. |
| I7 | MCP server creates new httpx client per tool call | agentic_reviewer | Each relay_send, relay_read etc. opens and closes a connection. Wasteful and adds latency. Should use a persistent client. Confirmed overlap with Cycle 1 item 9. |

## Implementation Notes

- **C1 + C4** are the highest-priority backend fixes. C4 overlaps with Cycle 1 item 10 - fix both with a single async-session refactor.
- **C2** (turn timeout) is the top reliability requirement for production multi-agent use. Can be implemented as a configurable TTL per turn with an auto-advance endpoint.
- **C3** is a quick fix: wrap callback props in useRef or useCallback in RelayDashboard.
- **C5** is a quick fix: add a single ErrorBoundary wrapper around routes in App.jsx.
- **I3** (wait_for_turn) was also identified in Cycle 1 as item 14 and has been elevated to IMPORTANT based on agentic review.
- **Cycle 1 P0 security items (1-4)** remain unfixed and should be addressed alongside these new findings.
